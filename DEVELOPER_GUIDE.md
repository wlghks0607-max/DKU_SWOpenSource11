# 개발자 가이드 (Developer Guide)

**점프 플랫포머 게임 — 물리 엔진 학습을 위한 오픈소스 2D 플랫포머**

이 문서는 프로젝트를 내려받아 직접 빌드·수정·확장하려는 개발자를 위한 안내서입니다. 이 게임은 점프 플랫포머의 기초 옵션을 모듈화하여, 누구나 쉽게 변형·가공하고 게임 물리 개념을 학습할 수 있도록 만드는 것을 목표로 합니다.

---

## 1. 개발 환경 요구사항

| 항목 | 요구사항 |
|------|----------|
| 게임 엔진 | Unity 6000.0.37f1
| 언어 | C# |
| IDE | VSCode
| 버전 관리 | Git / GitHub |

> Unity는 명시된 버전과 동일한 버전을 권장합니다. 버전이 다르면 프로젝트 임포트 시 자동 변환이 일어나며, 일부 설정이 달라질 수 있습니다.

---

## 2. 프로젝트 받기 / 실행

### 2-1. 저장소 클론 (데스크탑 사용 권장)

```bash
git clone _저장소_URL_
cd _프로젝트_폴더명_
```

### 2-2. Unity에서 열기

1. **Unity Hub**를 실행합니다.
2. **Add → Add project from disk** 로 클론한 폴더를 추가합니다.
3. 요구 Unity 버전으로 프로젝트를 엽니다.

---
## 3. 프로젝트 구조


```
Assets/
├── Scenes/            # 메인 메뉴, 각 레벨 씬
├── Scripts/
│   ├── Player.cs/        # 플레이어 이동·점프·상태 처리
│   ├── SceneChanger.cs/      # 씬 처리
│   ├── Goal.cs/            # 목표 지점 처리
│   └── Monster.cs/       # 몹 AI 처리
│   └── CameraFollow.cs/       # 카메라 처리
│   └── DieZone.cs/       # 낙사구간 설정
│   └── GameOverUi.cs/       # 몹 AI 처리

```

## 4. 핵심 아키텍처

게임은 입력 → 물리 → 충돌 → 상태 판정 순으로 동작합니다. 플레이어는 다음과 같은 **물리 상태**를 순환하며, 그 결과에 따라 게임 오버/클리어로 전이합니다. (자세한 흐름은 보고서의 *상태 다이어그램* 참고)


## 5. 커스터마이징 / 확장 가이드

이 프로젝트의 핵심 목적은 **쉽게 변형·확장할 수 있는 오픈소스 예제**가 되는 것입니다. 자주 바꾸게 될 항목을 정리합니다.

### 5-1. 물리 느낌 조정

플레이어 오브젝트의 인스펙터 또는 `PlayerController`의 공개 변수에서 다음 값을 조정해 조작감을 바꿀 수 있습니다.

| 변수 | 효과 | 비고 |
|------|------|------|
| `moveSpeed` | 최대 수평 이동 속도 | 클수록 빠르게 달림 |
| `runSpeed` | 가속도 | 클수록 빠르게 이동 |
| `jumpPower` | 점프 세기 | 클수록 높이 점프 |
| `fallMultiplier` | 중력 배율 (Rigidbody2D) | 클수록 빠르게 낙하, 점프가 무거워짐 |

> 점프 높이를 유지하면서 더 "쫀득한" 낙하를 원하면, 상승 중과 하강 중의 중력 배율을 다르게 적용하는 기법(가변 중력)을 고려할 수 있습니다.

### 5-2. 새 레벨 추가하기

1. 기존 레벨 씬을 복제합니다. (예: `Level1.unity` → `Level2.unity`_)
2. 발판·장애물·목표 지점을 재배치해 난이도를 구성합니다. (tilemap editor를 써서 맵 개발 가능)
3. `File → Build Settings`의 **Scenes In Build**에 새 씬을 추가합니다.
4. 레벨 선택 메뉴 / 레벨 목록 데이터에 새 레벨을 등록합니다.

## 6. 협업 / 기여 방법 (Git 워크플로)

본 팀은 Git을 통해 각자 작업하고 매주 통합하는 방식으로 진행했습니다. 외부 기여자도 동일한 흐름을 권장합니다.

```bash
# 1. 새 작업 브랜치 생성
git checkout -b feature/작업명

# 2. 작업 후 커밋
git add .
git commit -m "feat: 작업 내용 요약"

# 3. 원격에 푸시
git push origin feature/작업명

# 4. GitHub에서 Pull Request 생성 → 리뷰 → 병합
```

- 충돌(conflict)을 줄이기 위해 작업 단위를 작게 나누고 자주 통합합니다.
- 커밋 메시지는 변경 의도가 드러나게 작성합니다. (🟡 _커밋 컨벤션 합의 시 기입_)
